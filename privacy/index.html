Sirran

Privacy Policy

Effective date: 18 July 2026

App: Sirran (iPhone, Apple Watch, Android phone, Wear OS)

Publisher: Baris Demircan

Contact: barisdemircan@outlook.com.tr

Sirran has no user accounts, advertising SDK, advertising identifier, or first-party analytics service. Prayer calculations, Qibla, bundled Quran reading, and tasbih data are processed on your device. Optional Nearby Mosques, official prayer-timetable, store, and Quran Foundation online-content features make only the limited requests described below.

What the publisher does not collect directly

  • We do not operate a backend account system or sign-in.
  • We do not upload or retain a Quran reading-history profile, saved bookmarks, notes, prayer log, tasbih count, voice recordings, zakat figures, calendar entries, or keyboard input on a publisher-operated server. A user-invoked QF online-content request sends only the requested public-content identifier and ordinary connection data through the planned proxy, as described below.
  • We do not sell personal data. Third parties receive data only through the user-invoked network and store-purchase flows described below.
  • We do not use user data to build advertising profiles; sell, rent, mine, or repurpose Quran Foundation user data; or use notes, reflections, or other user-generated content to train artificial-intelligence models without the user's prior, separate, and explicit consent.

No advertising or analytics SDK

The Android, iPhone, Apple Watch, and Wear OS builds contain no advertising SDK, analytics SDK, or advertising-ID permission. Sirran Plus supports independent development; it does not remove advertising because advertising is absent for everyone.

Quran Foundation ecosystem and Content APIs

Sirran is an independent product and is not an official application of, affiliated with, or endorsed by Quran Foundation. Sirran's optional online Quran-content feature is designed to use the Quran Foundation Content APIs to retrieve public Quran text, translations, tafsir, recitation audio, timing, and related metadata from the Quran.com ecosystem. QuranReflect is part of Quran Foundation's wider ecosystem; Sirran does not access QuranReflect accounts, private reflections, bookmarks, notes, reading history, or other Quran Foundation user data.

This integration remains disabled until Quran Foundation grants access and the production security controls described below are in place. Sirran does not use end-user QF OAuth/OIDC, sign users into Quran Foundation accounts, or request user-data scopes. The planned backend will use the OAuth2 client_credentials flow to obtain an application access token with the minimum Content API scope; it will not receive end-user OAuth tokens. A public-content request identifies only the requested surah, ayah, recitation, translation, or tafsir resource. Sirran does not intentionally attach precise location, an advertising identifier, contact information, prayer history, bookmarks, notes, or a reading-history profile. Normal internet delivery still exposes ordinary connection data, such as IP address and request time, to the Sirran-controlled proxy and its hosting provider.

The planned proxy is intended to be hosted on Cloudflare Workers so Quran Foundation API credentials remain server-side and are never embedded in an app. The proxy may temporarily cache public QF content and metadata for no longer than seven days under standard access. Long-term offline QF recitation downloads remain disabled unless Quran Foundation grants express written permission and any required commercial/content licence. This policy will be updated before any QF OAuth, account, bookmark, note, reflection, history-sync, or other QF user-data feature is released.

Sensitive religious information

Prayer preferences, reading or recitation history, notes, reflections, worship logs, tasbih counts, and similar data may reveal religious beliefs and can be sensitive personal data. Sirran keeps saved versions of these items on the device and does not create or retain a user-linked reading-history or worship profile on a publisher server. An opted-in QF public-content request transiently processes the requested resource identifier and ordinary connection data as described above. If a future feature needs publisher-server collection or synchronization of saved religious information, it will remain off until the user gives a separate, explicit, affirmative opt-in that is not bundled into general acceptance of the Terms of Use. Data will then be limited to the disclosed feature and purpose.

Before the QF-powered online-content feature is enabled, Sirran will present a separate consent screen that explains the requested content identifiers and ordinary connection data sent through the proxy. The feature will make no QF request until the user takes an affirmative opt-in action. The user will be able to withdraw that consent in Settings, which disables future QF requests and clears device-local QF content. The shared proxy cache contains only public content, is keyed by content resource rather than a user or device, and expires within the standard seven-day limit. The feature remains disabled until this consent control is implemented and tested.

Nearby Mosques (optional, Android only)

When you open Nearby Mosques, Sirran rounds the device coordinate to four decimal places (about 11 metres of latitude resolution) and sends it to public OpenStreetMap Overpass instances to find mosques progressively within 2 km, 5 km, and at most 10 km. Google Play classifies this four-decimal coordinate as Precise location, even though it is rounded. The app may try more than one public instance when a server is unavailable. This happens only when you use that feature. The publisher does not store the coordinate or the results on a server. On the phone, an in-process cache keeps the four-decimal coordinate key, search radius, result list, and cache time. A result is treated as fresh for 10 minutes and may be shown as explicitly stale after a network failure for up to 24 hours. This cache is not written to disk and disappears when the app process ends. There is no Sirran account. Tapping a result opens your maps app for directions.

Official prayer timetable (optional)

The default calculation works locally. If you explicitly select an official timetable and confirm a supported place, Android sends only that public place identifier, the requested date range, and its time zone to the configured DeenMate timetable adapter over HTTPS. It does not send your GNSS coordinate, prayer history, Quran activity, microphone audio, or an advertising identifier. Authority credentials are never embedded in the app. Hosting and network providers necessarily process connection data such as an IP address to deliver an internet response.

Store purchases

On Android, Google Play Billing is used to offer and restore Sirran Plus. Google handles account and payment information under its own terms. Sirran stores the resulting entitlement locally and does not receive or store your payment-card details. The current iPhone launch configuration has in-app purchases disabled and is submitted as a free app with no in-app purchases; StoreKit support in the code is inactive in that build.

Third-party services and recipients

Sirran does not disclose worship activity for advertising or unrelated profiling. The following services may receive only the information needed for the user-requested feature:

  • Quran Foundation / Quran.com (planned, disabled pending approval): the Sirran proxy requests public Quran content using resource identifiers. See Quran.com Privacy.
  • Cloudflare (planned QF content proxy; optional timetable adapter when enabled): processes ordinary connection/request metadata and temporarily hosted content needed to deliver the response. See the Cloudflare Privacy Policy.
  • Public OpenStreetMap Overpass operators (Nearby Mosques only): receive the rounded coordinate and ordinary connection data described above. Public instances may be operated independently. See the OpenStreetMap Foundation Privacy Policy and the policy published by the selected instance, where available.
  • Google (Android/Wear OS): operates Google Play Billing, the Play store, and the Wear OS Data Layer described in this policy. See the Google Privacy Policy.
  • Apple (iPhone/Apple Watch): operates the App Store and the operating-system services used for permissions and paired-device connectivity. See the Apple Privacy Policy.
  • GitHub Pages: hosts this public policy and support site. Visiting the site is subject to the GitHub Privacy Statement.

Where a provider processes personal data on Sirran's behalf, the corresponding feature will be enabled only with appropriate data-protection terms and instructions that prohibit unrelated use and misuse of Quranic content. Public endpoints that act independently are recipients/controllers under their own policies rather than Sirran subprocessors. Sirran does not send Quran Foundation user data to OpenStreetMap, Google, Apple, or GitHub for unrelated use. The QF-powered feature stays disabled until any required legally binding processor terms and Islamic-content protection instructions have been verified.

On-device data and permissions

The app uses the following only on your device, to provide its features:

  • Location (optional): Used on your device to calculate accurate prayer times and the Qibla direction. If you do not grant location permission, you may explicitly select a fixed city; the app does not silently substitute one. A validated recent fix may be cached on your device for up to six hours so alarms, widgets, and the watch use the same location; it is not stored on a publisher server. On Android, location can leave the phone in two explicit product paths: a four-decimal coordinate is sent to Overpass when you open Nearby Mosques, and the exact effective phone coordinate can be synchronized to your paired Wear OS watch as described below. Selecting an official timetable sends the confirmed public place identifier described above, not the GNSS coordinate.
  • Notifications (optional): Used to deliver local prayer-time reminders scheduled on your device. No notification content is sent to a server.
  • Voice Tasbih microphone and speech recognition (optional, off by default, iPhone and Android): Voice Tasbih starts only after your opt-in and the system permission. Recognition runs on the device only; the feature is disabled when on-device recognition is not available and never falls back to network recognition. Its live recognition audio is not saved or transmitted off your device.
  • In Your Voice recordings (optional, Android only): When you explicitly start a recording in the Voice Library, Sirran saves the resulting .m4a file in the app's private local storage. You can play, manage, and delete these recordings from the library. They are not uploaded, synced, or sent to a third party.
  • Qibla camera preview (optional, Android only): If you open Qibla Horizon and grant camera permission, Sirran displays a live preview behind the Qibla overlay. The app has no photo/video capture or image-analysis use case and does not record, save, or transmit the preview.
  • Fajr Guardian activity recognition (optional, Android only): If you enable the walk-to-dismiss option and grant activity-recognition permission, Sirran counts step-detector events while the alarm screen is open. These events are used only to dismiss that alarm and are not saved or transmitted.
  • Wear OS rakah motion assistant (optional): While its screen is active, the watch processes accelerometer readings and, when available, gravity and gyroscope readings in memory. Raw motion samples are not saved or sent through the Wear Data Layer. Only the local session count, target, status, and save time are stored in the watch app's private preferences so an interrupted session can resume.
  • Do Not Disturb access (optional, off by default, Android): Used solely for the opt-in Quiet Hours feature, which silences your phone around each prayer and restores it afterwards. This is a device setting change only; no information is read, stored, or transmitted.
  • Calendar (optional, off by default): if you enable "prayer times in your calendar", the app writes prayer events into a separate, device-local calendar that never syncs to any account. Turning the feature off removes the events. The app does not read your other calendars' contents in this version.
  • Existing local Zakat Vault data (Android): If figures were entered in a prior or reviewed build, they remain only on your device in an encrypted file protected by the phone's hardware keystore. The current release UI is hidden pending religious review. The figures are never transmitted, and there is no Zakat account or sync service.
  • Sirran Keyboard (optional, Android): if you enable the bundled phrase keyboard in system settings, it only inserts the phrase you tap into the text field. It does not read, log, store, or transmit anything you type.
  • Apple Watch sync: Prayer preferences and tasbih state are shared between your iPhone and Apple Watch using Apple's device connectivity. Sirran does not send that data to a publisher server.
  • Wear OS Data Layer sync: The Android phone sends persistent Google Play services Data Items to paired Wear OS devices. These can include the exact effective phone coordinate, its accuracy and capture time (capturedAt), selected location/city, time zone, prayer method and madhab, tasbih label/count/target, remaining qada total, and active wird label/target. A Data Item is persistent until it is replaced or deleted and can be buffered while devices are disconnected. Google's Data Layer security overview and client-type reference state that only the same package signed with the same certificate can access the channel; direct Bluetooth uses standard Bluetooth encryption, and cloud-routed transfers are end-to-end encrypted. Google Play services may use Google-owned servers when Bluetooth is unavailable. Sirran does not operate a server for this sync.

Bundled Quran content

The Quran text in the app is "Tanzil Quran Text (Simple)" from the Tanzil Project (tanzil.net), used under the Creative Commons Attribution 3.0 license with in-app attribution. The bundled translation is "The Meaning of the Glorious Koran" by Marmaduke Pickthall (1930, English), which is in the public domain and shown with in-app attribution. No Turkish meal is currently bundled, so Turkish device locales show the Arabic Quran text without a meal. Opening and reading this bundled corpus does not make a network request. Optional online Quran Foundation content, when approved and enabled, follows the separate data flow and seven-day cache limit described above.

Sirran preserves Quran text unchanged, presents excerpts in context, and shows source attribution. QF content is not used for advertising profiles, biometric identification, or machine-learning training.

Children's privacy

Sirran is not directed to children under 13. It does not ask for a date of birth, has no sign-up flow, and therefore cannot accept an underage account registration. It does not knowingly collect children's personal data. Parents and guardians should supervise a child's use of religious content and optional network features.

Data security and retention

Sirran operates no first-party account or worship-history server. App settings (such as your selected city, madhab, and tasbih counts) are stored locally and are removed when you delete them in the app, clear app storage, or uninstall the app. The Nearby Mosques process cache has the 10-minute fresh/24-hour stale limits described above. Wear OS Data Items have Google Play services' persistent lifecycle described above and are overwritten when Sirran publishes newer state; they are not a publisher-server copy.

All Sirran-controlled network endpoints, including the planned QF proxy, must require TLS 1.2 or later. Before QF access is enabled, QF credentials must remain only in the server secret store, access must be least-privilege, temporary server caches must be encrypted at rest by the hosting platform, and production secrets must rotate at least every 90 days and immediately after suspected exposure. These controls are aligned with the QF Developer Privacy Packet's Security Rule 6.9. Sirran will begin responding to an actual or suspected API security incident in less than 24 hours and will report it to Quran Foundation no later than 24 hours after detection when QF systems or data may be affected.

Before activation, the proxy must disable application-level request-history logging or configure it so any IP address/resource-identifier combination is minimized and retained for no longer than seven days for security and abuse prevention. It must not add a Sirran user ID, device ID, advertising ID, or reading-history identifier to QF requests or logs.

Under standard QF access, temporary QF content is deleted or refreshed within seven days. Sirran receives no QF account/user data and therefore has no QF user-data backup today. If a future server feature stores user data, a verified deletion request will hard-delete it from live systems within 30 days and from backups within 90 days, unless a shorter legal period or lawful retention duty applies. Other requests are handled under each recipient's published retention and security practices.

Your rights and controls

You can deny or revoke optional operating-system permissions, disable optional features, delete feature data using the available in-app controls, clear app storage, or uninstall Sirran. Depending on applicable law, you may ask to access, correct, delete, restrict, object to, or obtain a copy of personal data controlled by the publisher. Send a verified request to the contact below; the publisher will respond within 30 days.

Sirran does not use end-user QF OAuth/OIDC or associate local data with a QF account. The planned backend application token is not an end-user token, so there is currently no user-granted Sirran QF token to revoke, and deleting a Quran Foundation account does not leave linked Sirran server data. End-user revocation controls, an OAuth revocation link, and an in-app server-data deletion mechanism will be added before any QF account/user-data integration ships.

International data transfers

Optional network providers may process ordinary connection data outside your country. Where Sirran selects a processor and cross-border transfer safeguards are legally required, Sirran will rely on the European Commission's Standard Contractual Clauses or an equivalent lawful mechanism and will comply with applicable local transfer laws. Independently operated public endpoints and app stores process data under their own policies and legal roles.
